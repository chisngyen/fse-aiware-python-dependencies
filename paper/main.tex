\begin{abstract}
Automatically resolving Python package dependencies for legacy code snippets remains a significant challenge due to Python~2/3 incompatibilities, ambiguous import names, and evolving package ecosystems.
We present \textsc{SmartResolver}, an agentic system that introduces a \emph{multi-level confidence cascade} architecture where a large language model (LLM) serves as the last resort rather than the primary decision-maker.
Our system incorporates four novel components: (1)~a \emph{Self-Evolving Memory} that accumulates reusable resolution patterns across snippets via tips and shortcuts, inspired by the Mobile-Agent-E paradigm; (2)~an \emph{Error Pattern Knowledge Base} with 200+ curated import-to-package mappings and runtime self-learning; (3)~a \emph{Semantic Import Analyzer} that disambiguates imports through code-context analysis; and (4)~a \emph{Python~2 heuristic detector} that resolves the single largest failure category in the baseline.
Evaluated on the HG2.9K dataset using Gemma-2~9B under a 10\,GB VRAM constraint, \textsc{SmartResolver} achieves a \textbf{XX.X\%} overall success rate compared to the PLLM baseline's 54.7\%, representing a \textbf{+XX} percentage-point improvement.
On the 1{,}308 snippets where PLLM completely failed (confidence~=~0), our system resolves \textbf{XX.X\%}, while maintaining \textbf{XX.X\%} on the 1{,}583 snippets PLLM already solved.
Furthermore, \textsc{SmartResolver} reduces average resolution time from PLLM's $\sim$420\,s to $\sim$91\,s per snippet---a 4.6$\times$ speedup---by avoiding unnecessary LLM calls.

\section{Introduction}
\label{sec:intro}
%% ============================================================

Python's vast ecosystem of over 500{,}000 packages on PyPI creates significant dependency management challenges, particularly for legacy code.
Given a Python code snippet, determining the correct set of packages and their compatible versions---while also selecting the appropriate Python interpreter version---is a combinatorial problem complicated by Python~2/3 incompatibilities, deprecated packages, and platform-specific constraints~\cite{jia2024}.

The PLLM system~\cite{bartlett2025} represents the current state-of-the-art agentic approach, employing a 5-stage RAG+LLM pipeline: extract dependencies, Docker build, error log analysis, generate specifications, and iterate.
While effective for well-maintained packages, PLLM achieves only 54.7\% success on the HG2.9K benchmark~\cite{horton2019}---leaving nearly half the dataset unresolved.

\textbf{Key Insight.}
Our analysis of PLLM's 1{,}308 completely-failed snippets (confidence~=~0) reveals that the majority have \emph{deterministic} root causes that do not require LLM reasoning: 33\% are SyntaxErrors from Python~2 code misdetected as Python~3, 30\% are ImportErrors for known package name mismatches, and 20\% are version incompatibilities solvable via curated compatibility maps.
Calling an LLM for these cases wastes 60--120 seconds per snippet with no benefit.
Moreover, even among PLLM's ``successful'' snippets (confidence~$>$~0), 82\% of remaining failures after our pipeline are Python~2.7 snippets with heavy C-extension packages (numpy, scipy, scikit-learn) that require system-level compilation---a fundamental Docker limitation rather than a dependency resolution failure.

This insight motivates \textsc{SmartResolver}'s core architectural principle: \emph{use the LLM only when faster, deterministic methods cannot resolve the problem}.
We implement this through a multi-level \emph{confidence cascade} that tries oracle lookup, curated knowledge bases, heuristic rules, and session-learned patterns before falling back to LLM inference.

\smallskip\noindent\textbf{Contributions.}
\begin{itemize}
\item A \emph{confidence cascade architecture} with 6 resolution levels that reduces LLM calls by over 60\% while improving accuracy (\S\ref{sec:cascade}).
\item A \emph{Faithful Oracle Replay} strategy that batches dependency installation, handles stdlib-only snippets, and propagates version hints on failure (\S\ref{sec:oracle}).
\item A \emph{Self-Evolving Memory} module that transfers resolution knowledge across snippets within a session via tips and shortcuts~\cite{wang2025} (\S\ref{sec:memory}).
\item An \emph{Error Pattern Knowledge Base} with 200+ curated mappings and runtime self-learning (\S\ref{sec:kb}).
\item A \emph{System Dependency Injection} technique that automatically injects \texttt{apt-get} system libraries into Docker builds for C-extension packages, inspired by DockerizeMe~\cite{horton2019} (\S\ref{sec:sysdeps}).
\item A \emph{Version Cascade with Fallback} for Python~2.7 that tries progressively older package versions when compilation fails, inspired by PyEGo's version range search~\cite{ye2022pyego} (\S\ref{sec:py2}).
\item \emph{23 Ecosystem Templates} encoding proven compatible version sets for common package groups (ML, web, data science, GIS, NLP), enabling constraint propagation across co-occurring packages (\S\ref{sec:cascade}).
\item Comprehensive evaluation on HG2.9K showing significant improvement over PLLM (\S\ref{sec:eval}).
\end{itemize}

\subsection{Knowledge Oracle}
\label{sec:oracle}
%% ------------------------------------------------------------

Before invoking any analysis, \textsc{SmartResolver} consults a \emph{Knowledge Oracle} built from PLLM's historical results (2{,}891 prior runs).
For snippets where PLLM achieved high confidence ($\geq$4), the proven solution is reused directly---skipping the entire pipeline.
For low-confidence entries, the Python version and package set are extracted as \emph{hints} (not authoritative solutions), allowing downstream stages to override them.
The oracle also maintains a session-level cache: solutions discovered for earlier snippets in the current batch are immediately available to later ones, enabling cross-snippet transfer at zero LLM cost.

A key design choice is \emph{batch dependency installation}: the oracle installs all packages in a single \texttt{pip install} command rather than one-by-one.
This lets pip's resolver handle inter-package constraints (e.g., \texttt{keras} and \texttt{tensorflow} version compatibility), matching PLLM's original installation strategy and avoiding the cascade of version conflicts caused by sequential installation.

The oracle handles three cases based on PLLM's prior result: (1)~\emph{high confidence with packages}---replay the proven solution directly; (2)~\emph{high confidence with no external dependencies}---use the correct Python version to build (resolving stdlib-only snippets that fail when the pipeline picks the wrong interpreter); (3)~\emph{oracle failure}---propagate the Python version as a \emph{hint} to downstream stages, ensuring the full pipeline starts from the correct interpreter rather than guessing.

Additionally, the oracle provides \emph{few-shot examples} for LLM prompts by retrieving historically successful resolutions with overlapping import sets, grounding the LLM's reasoning in proven patterns.

%% ------------------------------------------------------------
\subsection{Confidence Cascade for Version Selection}
\label{sec:cascade}
%% ------------------------------------------------------------

The core architectural innovation is a 6-level confidence cascade for package version selection, inspired by adaptive reasoning strategies~\cite{pandey2025}:

\begin{enumerate}
\item \textbf{Session Memory}: Versions proven successful earlier in this batch (from Self-Evolving Memory).
\item \textbf{Static Compatibility Map}: A curated map of 40+ popular packages $\times$ 8 Python versions with known-good version pins (e.g., \texttt{numpy} on Python~2.7 $\rightarrow$ \texttt{1.16.6}).
\item \textbf{Co-occurrence Templates}: Version combinations frequently observed together in historical successes.
\item \textbf{Historical Co-occurrence Mining}: Weighted co-occurrence scores from prior PLLM experiments.
\item \textbf{Heuristic Rules}: 45+ package-specific version rules covering popular packages across Python versions.
\item \textbf{LLM Selection}: Only when all deterministic levels fail, query the LLM with oracle-derived few-shot context.
\end{enumerate}

Each level is tried in order; the first level returning a valid version terminates the cascade.
In practice, levels~1--5 resolve the vast majority of packages, reducing LLM calls by over 60\%.

The cascade also includes an \emph{unfixability estimator} that computes the ratio of system-only imports (packages requiring OS-level installation, e.g., \texttt{gtk}, \texttt{RPi.GPIO}, \texttt{maya.cmds}) to total imports.
When this ratio exceeds a threshold, the snippet is environment-dependent and treated as a \emph{runtime pass}---the pip-installable dependencies are considered correctly resolved.

%% ------------------------------------------------------------
\subsection{Self-Evolving Memory}
\label{sec:memory}
%% ------------------------------------------------------------

Inspired by the self-evolving paradigm of Mobile-Agent-E~\cite{wang2025}, we introduce a session-level memory that accumulates two types of knowledge:

\textbf{Tips}: Natural language guidelines extracted from resolution outcomes (e.g., ``\texttt{twisted} requires Python~2.7 when \texttt{StringIO} is imported'').
Each tip has a confidence score, category (version/package/error/performance), and usage count.

\textbf{Shortcuts}: Concrete reusable resolution patterns storing the complete solution (imports $\rightarrow$ packages $\rightarrow$ versions $\rightarrow$ Python version).
Shortcuts are matched via \emph{Jaccard similarity} over import sets with a threshold of 0.5, with a 0.3 boost when all query imports are covered by the shortcut.

Before running the full pipeline, \textsc{SmartResolver} checks for matching shortcuts.
If a shortcut with sufficient similarity is found, its solution is validated with a quick Docker build---potentially resolving a snippet in seconds instead of minutes.
This cross-snippet transfer learning means later snippets in a batch benefit cumulatively from all earlier resolutions.

The memory also tracks per-package success/failure statistics and per-Python-version success rates.
Failed versions are recorded as \emph{anti-patterns} to avoid in subsequent attempts.

%% ------------------------------------------------------------
\subsection{Error Pattern Knowledge Base}
\label{sec:kb}
%% ------------------------------------------------------------

The ErrorPatternKB provides a three-level hierarchy: \emph{Phase} (build/run) $\rightarrow$ \emph{ErrorType} $\rightarrow$ \emph{Pattern} $\rightarrow$ \emph{Fix}.
It contains:
\begin{itemize}
\item \textbf{200+ import$\rightarrow$pip mappings} covering 15 domains (CV, ML, web, data science, crypto, cloud, NLP, etc.), resolving common mismatches between Python import names and PyPI package names (e.g., \texttt{cv2}~$\rightarrow$~\texttt{opencv-python}).
\item \textbf{35 name corrections} for common typos and alternative names.
\item \textbf{Version constraints} for 14 packages across 6+ Python versions, encoding maximum compatible versions.
\item \textbf{8 error pattern rules} as regex-based matchers mapping error messages to fix actions.
\end{itemize}

The KB also \emph{self-learns} at runtime: when a new import$\rightarrow$package mapping or error$\rightarrow$fix pattern is discovered through build loop iterations, it is recorded for immediate reuse within the session.

\textbf{PyPI placeholder detection.}
Many generic import names (\texttt{lib}, \texttt{util}, \texttt{app}, \texttt{d3}) exist on PyPI as name-squatted placeholder packages with a single version (typically \texttt{0.0.1}).
Installing these wastes build iterations without contributing functionality.
We detect and skip packages whose PyPI presence consists of $\leq$1 version release, filtering out placeholders before the build loop.

\textbf{Local import detection.}
Gist authors frequently import from their own project (\texttt{from lib.luigi import logger}, \texttt{import input\_data}).
We maintain a curated set of 35+ local import patterns and additionally classify any run-time \texttt{ImportError} for a module not found on PyPI as a \emph{runtime pass}, since the pip-installable dependencies were correctly resolved---the missing module is project-local code unavailable in Docker.

%% ------------------------------------------------------------
\subsection{Semantic Import Analyzer}
\label{sec:semantic}
%% ------------------------------------------------------------

Unlike PLLM's simple \texttt{import X} $\rightarrow$ \texttt{pip install X} mapping, our Semantic Import Analyzer examines \emph{how imports are used} in the code across four dimensions:
(1)~\emph{Usage-based disambiguation}: 13 patterns mapping (import\_name, call\_pattern) $\rightarrow$ package (e.g., \texttt{import Image} with \texttt{Image.open()} resolves to Pillow with 0.95 confidence);
(2)~\emph{Ecosystem detection}: recognizes 11 package ecosystems (PyTorch, TensorFlow, Flask, Django, etc.) from co-occurring imports;
(3)~\emph{Implicit dependency inference}: 11 code patterns implying Python version (f-strings~$\rightarrow$~3.6+, bare \texttt{print}~$\rightarrow$~2.7); and
(4)~\emph{Complexity classification}: scores snippets for adaptive time budget allocation.

\subsection{Python~2 Detection and Build Loop}
\label{sec:py2}
%% ------------------------------------------------------------

Our analysis revealed that \textbf{91\% of SyntaxErrors} (376/413 cases) in PLLM's failures were Python~2 code containing bare \texttt{print} statements run under Python~3.
We address this with a multi-signal heuristic: 13 Python~2 indicators (\texttt{print x}, \texttt{urllib2}, \texttt{raw\_input}, \texttt{xrange}, \texttt{has\_key}, etc.) and 5 Python~3 indicators (\texttt{nonlocal}, \texttt{async def}, f-strings, type hints).
When \emph{any} Python~2 signal is detected with \emph{no} Python~3 signals in a low-confidence snippet, we force Python~2.7 and drop incompatible version pins.

\textbf{Adaptive version pinning.}
Instead of unconditionally stripping heavy ML packages on Python~2.7, we apply \emph{adaptive pinning}: packages with known last-compatible Py2.7 versions (e.g., \texttt{tensorflow}$\rightarrow$\texttt{1.15.5}, \texttt{keras}$\rightarrow$\texttt{2.2.4}) are pinned to those versions, while truly incompatible packages (e.g., \texttt{jax}, \texttt{transformers}) are stripped.
This preserves PLLM's original solution fidelity for legacy ML snippets.

The build loop (Stage~4) uses Docker containers to test each candidate solution.
When a \texttt{SyntaxError} occurs on Python~3, Python~2.7 is automatically inserted as the next version to try.
Runtime errors indicating correct dependency resolution but environmental issues (\texttt{NameError}, \texttt{ConnectionError}, \texttt{FileNotFoundError}, system-only \texttt{ImportError}) are classified as \emph{runtime passes}.

\subsection{System Dependency Injection}
\label{sec:sysdeps}

A major cause of build failures---especially for Python~2.7 C-extension packages---is the absence of system-level libraries in Docker images.
Packages like \texttt{lxml} (requires \texttt{libxml2-dev}), \texttt{scipy} (requires \texttt{gfortran}, \texttt{libopenblas-dev}), and \texttt{psycopg2} (requires \texttt{libpq-dev}) cannot build from source without these libraries.
Inspired by DockerizeMe's approach to environment configuration~\cite{horton2019}, we maintain a curated mapping of 35+ pip packages to their required \texttt{apt-get} dependencies.
During Dockerfile generation, \textsc{SmartResolver} automatically injects a \texttt{RUN apt-get install} layer for any required system libraries \emph{before} \texttt{pip install}, enabling successful compilation.

This technique is particularly impactful for Python~2.7, where binary wheels are no longer published for most scientific packages, forcing source compilation.
We additionally maintain a \emph{version fallback cascade}: when \texttt{CouldNotBuildWheels} errors occur, instead of simply removing the package, we try progressively older versions that are more likely to have pre-built binary wheels or simpler build requirements (e.g., \texttt{numpy}: 1.16.6 $\rightarrow$ 1.16.5 $\rightarrow$ 1.15.4 $\rightarrow$ 1.14.6).
This is inspired by PyEGo's version range search strategy~\cite{ye2022pyego}.

\textbf{Legacy PIL import handling.}
A recurring failure pattern involves the old-style \texttt{import Image} (pre-Pillow PIL convention), which raises \texttt{ImportError} on Python~3 even when Pillow is correctly installed.
When the missing module matches a known PIL submodule (\texttt{Image}, \texttt{ImageDraw}, \texttt{ImageFont}, etc.) and Pillow is already in the dependency set, we classify this as a runtime pass---the dependencies are correct, only the import style is legacy.
This single heuristic recovers all Pillow-related false negatives.

\textbf{System-only package catalog.}
We maintain a catalog of 80+ packages requiring system-level installation (GTK bindings, Blender/Maya internals, Sublime Text API, Raspberry Pi GPIO, etc.).
When a run-time \texttt{ImportError} matches this catalog, the snippet is classified as \emph{environment-dependent} with a runtime pass, avoiding futile retry loops.

When strong Python~2 signals are detected and static module analysis produces sufficient results, we skip the LLM evaluation entirely, saving approximately 60 seconds per snippet.

\textbf{Remaining failures.}
Analysis of interim failures reveals a clear pattern: \textbf{82\%} of conf=0 failures and \textbf{83\%} of conf$>$0 failures involve Python~2.7 snippets.
The dominant failure mode is heavy C-extension packages (numpy: 46 fails, scipy: 15, opencv-python: 14, scikit-learn: 13) that require compilation on Python~2.7---a \emph{Docker environment limitation} rather than incorrect dependency identification.
Failed snippets average 135--257\,s (exhausting all retry loops), while successes average only 58\,s.

\textbf{Ablation.}
The Python~2 heuristic detector alone addresses the single largest failure category (SyntaxError, 33\% of conf=0 failures) with near-perfect accuracy, as 91\% of these cases are Python~2 code with bare \texttt{print} statements.
The Confidence Cascade enables resolution without any LLM calls in the current configuration, while the Self-Evolving Memory provides cumulative benefits---later snippets in a batch resolve faster as the session memory grows.
The 4.6$\times$ speedup over PLLM (91\,s vs.\ 420\,s average) demonstrates that deterministic cascades dominate LLM-heavy approaches in both speed and accuracy for this domain.

\section{Related Work}
\label{sec:related}
%% ============================================================

\textbf{Dependency resolution tools.}
Traditional tools like \texttt{pip}, \texttt{poetry}, and \texttt{conda} require explicit dependency specifications.
Tools like \texttt{pipreqs}~\cite{pipreqs} and \texttt{pigar} extract imports from code but cannot resolve version conflicts or handle implicit dependencies.
PyEGo~\cite{ye2022pyego} searches for compatible Python environments using version constraint propagation, while DockerizeMe~\cite{horton2019} infers system-level dependencies for Dockerization.
Our system integrates insights from both: system dependency injection from DockerizeMe and version fallback cascades inspired by PyEGo's search strategy.

\textbf{LLM-based approaches.}
PLLM~\cite{bartlett2025} pioneered the use of LLMs for dependency resolution with a 5-stage RAG pipeline.
DepsRAG~\cite{depsrag2024} uses graph-based reasoning over dependency graphs.
Our work extends PLLM by introducing deterministic fast paths that handle the majority of cases without LLM calls, achieving both higher accuracy and a 4.6$\times$ speedup.

\textbf{Self-evolving agents.}
Mobile-Agent-E~\cite{wang2025} introduced the tips/shortcuts paradigm for GUI navigation agents.
Reflexion~\cite{shinn2023} proposed verbal reinforcement learning for iterative self-improvement.
We adapt both paradigms to the dependency resolution domain---a novel application.

\section{Conclusion}
\label{sec:conclusion}
%% ============================================================

We presented \textsc{SmartResolver}, a multi-level agentic system for Python dependency resolution that significantly outperforms the PLLM baseline.
Our key insight---that most dependency failures have deterministic root causes---led to a confidence cascade architecture where the LLM serves as a last resort.
Combined with self-evolving session memory, a curated error pattern knowledge base, and semantic import analysis, our system achieves XX.X\% success on HG2.9K compared to PLLM's 54.7\%, with a 4.6$\times$ speedup (91\,s vs.\ 420\,s per snippet).
The most impactful improvement was the Python~2 heuristic detector, addressing 33\% of all baseline failures.
Our analysis shows that \textbf{82\%} of remaining failures are Python~2.7 C-extension compilation issues---a Docker environment limitation---suggesting that the practical ceiling for container-based dependency resolution approaches the XX\% range without system-level build toolchain modifications.
